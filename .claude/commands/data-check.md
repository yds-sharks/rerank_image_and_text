# 数据状态检查

扫描各子模块的 data/ 目录，按以下数据分区规则和检查项进行报告。只读不写，只给建议，不执行操作。

---

## 数据分区规则

每个子模块的 `data/` 分为两个区域：

### tmp/（临时存储）
- 存放正在处理的中间结果
- 新一轮处理可覆盖旧文件
- **未经用户明确指令禁止删除**（即使看起来过时了，也要先问）
- 每次任务完成后必须提醒用户哪些文件可沉淀或清理

### persistent/（长期存储）
- 仅在用户明确确认后从 tmp/ 移入
- 视为项目资产，不可随意修改或删除
- 下设 `bad_cases/` 目录

### persistent/bad_cases/
- 记录处理中遇到的典型错误或边界case
- 每条记录包含：样本ID、问题描述、原因分析、处理方式
- 格式：JSON文件，按批次或主题组织

---

## 通用检查项

对每个子模块的 data/ 目录：

- tmp/ 中的文件、大小、修改时间
- persistent/ 中的文件
- bad_cases/ 目录数及各目录的 README 摘要
- 标记 tmp/ 中**超过3天未变动**的文件 → 提醒用户决策
- 标记明显的临时测试文件（`_test`、`_tmp` 后缀）→ 建议清理
- 待沉淀或待清理的建议

---

## 子模块专项检查

对有结构化输出的子模块（如评测、标注等），额外检查：

- 输出目录下文件完整性（是否缺少必要文件）
- 标记 tmp/ 中有输入但输出目录中无对应结果的（未归档）
- bad_cases/ 中各 case 的问题概述（读取 README.md 第一行）

---

## bad_cases 规范检查

- 每个 bad case 目录必须有 README.md
- 必须有可独立复现的单样本数据文件（非全表）
- 标记缺失 README 或缺少数据文件的 bad case
