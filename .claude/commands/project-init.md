# 项目结构初始化

根据以下规范创建项目结构。

$ARGUMENTS
- modules: 子模块名称列表（逗号分隔，可选）

## 初始化流程

1. 与用户确认子模块划分
2. 创建全部目录
3. 每个子模块生成 MODULE_SPEC.md（模板见下方）
4. 生成 README.md（项目目标 + 子模块表 + 数据流 + 目录树）
5. 生成 CLAUDE.md（精简，只放每次必知的信息）
6. 更新 memory

---

## 顶层结构规范

主目录**只做项目组织和文档管理**，不承担任何具体代码工作。

```
{项目根}/
├── README.md                # 全局说明：项目目标、子模块职责、数据流
├── CLAUDE.md                # Claude上下文（精简，自动加载）
├── .claude/commands/        # Skill触发器（轻量）
├── {子模块A}/
├── {子模块B}/
└── {子模块C}/
```

**主目录只放：** README.md、CLAUDE.md、.claude/、子模块目录
**主目录禁止：** 具体代码文件、config/、data/、logs/、scripts/、tmp/、shared/

README.md 第一节必须是**项目目标描述**：解决什么问题、服务什么场景、当前阶段重点。

---

## 子模块内部结构

```
{子模块}/
├── MODULE_SPEC.md           # 本模块规范
├── code/                    # 源代码
├── config/                  # 配置文件
├── data/
│   ├── tmp/                 # 临时存储（可覆盖不可擅删）
│   └── persistent/          # 长期存储（用户确认后移入）
│       └── bad_cases/       # 特殊case记录
├── instructions/            # 沉淀的处理指令文档
└── notes/                   # 项目笔记与操作记录
    └── ops/                 # 操作记录（方案 + 步骤日志）
```

---

## 操作记录规范（notes/ops/）

每个任务/阶段在 `notes/ops/` 下维护一套记录：

```
notes/ops/
├── {任务名}_plan.md          # 方案设计文档
└── {任务名}/                 # 步骤执行记录
    ├── step01_{步骤名}.md
    ├── step02_{步骤名}.md
    └── ...
```

### 方案设计文档（`{任务名}_plan.md`）

```markdown
# {任务名} 方案

## 目标
{要解决什么问题}

## 步骤规划

- [x] Step 1: {步骤名} — {一句话说明}
- [x] Step 2: {步骤名} — {一句话说明}
- [ ] Step 3: {步骤名} — {一句话说明}  ← 当前进度
- [ ] Step 4: {步骤名} — {一句话说明}

## 关键决策
- {决策内容} — 原因：{为什么这样做}

## 最终产出
- {产出文件路径和说明}
```

**进度标注规则：** `[x]` 已完成、`[ ]` 待执行、`← 当前进度` 标记下一步。每次开始工作前先读此文件确认进展。

### 步骤执行记录（`{任务名}/stepNN_{步骤名}.md`）

```markdown
# Step {N}: {步骤名}

## 执行内容
{做了什么}

## 中途调整
- {调整内容} — 原因：{为什么改}

## Bad Cases
- {样本ID}: {问题现象} → {处理方式}

## 产出
- {文件路径}: {说明}
```

### 行为要求

1. **开始工作前**：先读 `{任务名}_plan.md`，确认当前进度和下一步
2. **每步完成后**：更新 plan 中的进度标注（`[x]`），写对应 step 记录
3. **遇到调整**：在 step 记录中写明改了什么和为什么
4. **遇到 bad case**：在 step 记录中记录，同时考虑是否需要写入 `data/persistent/bad_cases/`

---

## 文件命名规则

所有文件以**子模块名为前缀**，避免跨模块冲突：
- 中间产物: `{模块}_{序号}_{步骤}.ext`（如 `gt_01_fused.json`）
- 最终产出: `{模块}_{语义名}.ext`（如 `eval_metrics.json`）
- 日志: `{模块}_{名称}_log.ext`
- 测试: `{模块}_test_{描述}.ext`

## MODULE_SPEC.md 模板

每个子模块根目录生成此文件：

```markdown
# {模块名} 模块规范

## 模块职责
{一句话说明}

## 目录结构

{模块名}/
├── MODULE_SPEC.md
├── code/
├── config/
├── data/
│   ├── tmp/
│   └── persistent/
│       └── bad_cases/
├── instructions/
└── notes/
    └── ops/                 # 操作记录（方案 + 步骤日志）

## 文件命名

所有文件以 `{模块名}_` 为前缀：
- 中间产物: `{模块名}_{序号}_{步骤}.ext`
- 最终产出: `{模块名}_{语义名}.ext`
- 日志: `{模块名}_{名称}_log.ext`

## 上下游关系
- 输入来源: {上游模块或外部}
- 输出去向: {下游模块或最终交付}

## 关键注意事项
{本模块特有的踩坑经验，随项目推进持续补充}
```

## 结构变更规则

1. **新增子模块** — 创建目录和 MODULE_SPEC.md → 更新 README.md 和 CLAUDE.md → 更新 memory
2. **调整现有结构** — 先与用户确认 → 执行后同步更新所有文档 → 在 notes/ 记录变更原因
3. **废弃子模块** — 先在 MODULE_SPEC.md 标注"已废弃"及原因 → 用户确认后再清理

变更完成后主动检查所有文档一致性。
